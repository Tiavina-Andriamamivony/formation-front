De HTML/CSS
vers React TypeScript

Tu connais déjà HTML et CSS. Ce module t'explique ce qui change vraiment quand on passe à React et pourquoi TypeScript est ton meilleur allié dès le départ.

Durée : ~2h
Niveau : Débutant React
Stack : React 19 + TS
01

Le vrai changement de paradigme

En HTML/CSS, tu penses en fichiers et en pages. En React, tu penses en composants — des morceaux d'UI réutilisables, isolés, avec leur propre état.

La différence fondamentale : HTML est déclaratif mais statique. React est déclaratif et dynamique — l'UI est une fonction de la donnée. Si la donnée change, l'UI se met à jour automatiquement.

HTML classique
<!-- Statique, tu dois JS pour tout -->
<button id="btn">Cliquer</button>
<p id="count">0</p>

<script>
  let n = 0
  document.getElementById('btn')
    .addEventListener('click', () => {
      n++
      document.getElementById('count')
        .textContent = n
    })
</script>
React + TypeScript
// L'UI suit la donnée automatiquement
import { useState } from 'react'

export function Counter() {
  const [n, setN] = useState(0)

  return (
    <div>
      <button onClick={() => setN(n + 1)}>
        Cliquer
      </button>
      <p>{n}</p>
    </div>
  )
}
À retenir

Tu n'écris plus document.getElementById pour modifier le DOM. Tu décris à quoi l'UI doit ressembler selon la donnée, et React se charge du reste.

02

JSX — pas du HTML, du JavaScript

JSX ressemble à du HTML mais c'est du JavaScript transformé par le compilateur. Il y a quelques règles à connaître dès le départ.

Différences clés HTML → JSX : ce qui change 
// HTML                     JSX
class="..."          →   className="..."
for="..."            →   htmlFor="..."
<br>                 →   <br />    (self-closing obligatoire)
<img src="...">      →   <img src="..." />
style="color:red"    →   style={{ color: 'red' }}

// Expressions JS avec { }
<h1>{user.name}</h1>
<div className={isActive ? 'active' : 'idle'}>

// Un seul élément racine
return (
  <>        {/* Fragment — pas de div inutile */}
    <h1>...</h1>
    <p>...</p>
  </>
)
03

Composants et props typées

Un composant React est une fonction qui retourne du JSX. TypeScript te permet de définir exactement ce que ce composant accepte — plus d'erreurs "undefined is not a function".

Exemple concret Carte service MA-ERI typée 
// ① On définit la forme des données (interface)
interface ServiceCardProps {
  titre: string
  description: string
  numero: number
  lien?: string  // ? = optionnel
}

// ② Le composant reçoit ces props typées
export function ServiceCard({ titre, description, numero, lien }: ServiceCardProps) {
  return (
    <div className="service-card">
      <span className="num">{numero.toString().padStart(2, '0')}</span>
      <h3>{titre}</h3>
      <p>{description}</p>
      {lien && <a href={lien}>Explorer →</a>}
    </div>
  )
}

// ③ Utilisation — TypeScript te prévient si tu oublies un prop requis
<ServiceCard
  numero={1}
  titre="Approvisionnement industriel"
  description="Roulements, courroies, joints..."
/>
Pourquoi TypeScript dès maintenant ?

L'interface ServiceCardProps est un contrat. Si tu utilises <ServiceCard /> sans titre, TypeScript te crie dessus avant même de lancer le projet. C'est 80% des bugs évités à la source.

React fonctionnel —
normes & concepts

useState, useEffect, les règles des hooks, la composition de composants, et les patterns qu'on utilise vraiment en prod.

Durée : ~2h
Niveau : Intermédiaire
Stack : React 19 + TS
01

useState — gérer l'état local

L'état, c'est une donnée qui appartient à un composant et qui, quand elle change, déclenche un re-render de l'UI. TypeScript te force à être précis sur le type dès l'initialisation.

Pattern Types courants avec useState 
// Primitifs — TS infère le type automatiquement
const [count, setCount] = useState(0)          // number
const [nom, setNom] = useState('')            // string
const [open, setOpen] = useState(false)      // boolean

// Objet — déclare l'interface
interface Produit {
  id: string
  nom: string
  prix: number
}
const [produit, setProduit] = useState<Produit | null>(null)

// Tableau typé
const [panier, setPanier] = useState<Produit[]>([])

// Mise à jour fonctionnelle (important !) — toujours utiliser
// la forme fonctionnelle quand le nouveau state dépend de l'ancien
setCount(prev => prev + 1)  // ✅ safe
setCount(count + 1)          // ⚠️  peut poser problème en async
Erreur classique

Ne jamais muter directement le state. panier.push(item) ne déclenche pas de re-render. Il faut toujours créer un nouveau tableau : setPanier(prev => [...prev, item]).

02

useEffect — effets de bord

useEffect sert à synchroniser un composant avec quelque chose d'extérieur : une API, le localStorage, un timer, un event listener. Ce n'est PAS fait pour la logique métier.

Patterns Les 3 formes de useEffect 
// 1. Exécuté à chaque render — rarement utile
useEffect(() => { console.log('render') })

// 2. Exécuté une seule fois au montage (équivalent componentDidMount)
useEffect(() => {
  fetchServices()
}, [])  // ← tableau de dépendances vide

// 3. Exécuté quand une valeur change
useEffect(() => {
  if (!userId) return
  fetchUserData(userId)
}, [userId])  // ← relancé à chaque changement de userId

// Cleanup — important pour les subscriptions, timers
useEffect(() => {
  const timer = setInterval(() => ..., 1000)
  return () => clearInterval(timer)  // nettoyage au démontage
}, [])
React 19 — note importante

Avec React 19, pour du data fetching simple, on préfère les Server Components (vu en formation 3) plutôt que useEffect. useEffect reste utile pour les interactions DOM, les timers, et les event listeners.

03

Les règles des hooks

Les hooks ont deux règles absolues. Les violer crée des bugs impossibles à debugger.

Toujours au top level — jamais dans un if, un for, ou une fonction imbriquée. React compte les hooks dans l'ordre à chaque render.
Seulement dans des composants React ou des custom hooks — jamais dans une fonction utilitaire normale.
Les custom hooks commencent par "use" — useCart, useServices. C'est une convention que ESLint vérifie automatiquement.
04

Custom hooks — extraire la logique

Un custom hook, c'est une fonction qui utilise d'autres hooks. L'objectif est de séparer la logique de l'affichage.

Exemple useCart — logique du panier e-commerce MA-ERI 
// hooks/useCart.ts
interface Article {
  id: string
  nom: string
  prix: number
  quantite: number
}

export function useCart() {
  const [articles, setArticles] = useState<Article[]>([])

  const ajouter = (article: Omit<Article, 'quantite'>) => {
    setArticles(prev => {
      const existant = prev.find(a => a.id === article.id)
      if (existant) {
        return prev.map(a => a.id === article.id
          ? { ...a, quantite: a.quantite + 1 } : a)
      }
      return [...prev, { ...article, quantite: 1 }]
    })
  }

  const total = articles.reduce(
    (acc, a) => acc + a.prix * a.quantite, 0
  )

  return { articles, ajouter, total }
}

// Dans le composant — clean, pas de logique
export function PanierPage() {
  const { articles, ajouter, total } = useCart()
  // ...
}

React vs Next.js —
pourquoi le saut ?

React c'est une librairie d'UI. Next.js c'est un framework complet bâti sur React. Ce module explique ce que Next.js résout et pourquoi c'est le choix standard en prod.

Durée : ~2h
Niveau : Intermédiaire
Stack : Next.js 15
01

Ce que React seul ne résout pas

React gère l'UI. Le reste — routing, performance, SEO, data fetching, déploiement — c'est à toi de l'assembler. Next.js fait tout ça par défaut.

React seul (Vite/CRA)
Routing à configurer manuellement (react-router)
Tout le JS envoyé au client (mauvais pour SEO)
Pas d'API routes — besoin d'un backend séparé
Images, fonts, cache : à configurer soi-même
Next.js 15
Routing basé sur les fichiers — zero config
Server Components — HTML rendu côté serveur, parfait pour le SEO
API Routes et Server Actions intégrés
next/image, next/font, cache optimisé out-of-the-box

03

Server vs Client Components

C'est le concept central de Next.js 15. Par défaut, tous tes composants sont des Server Components — ils s'exécutent sur le serveur, jamais dans le navigateur.

Règle Quand utiliser "use client" ? 
// Server Component (défaut) — données, SEO, pas d'état
// app/services/page.tsx
async function ServicesPage() {
  // fetch direct côté serveur — pas besoin d'useEffect !
  const services = await fetch('https://api.ma-eri.com/services')
    .then(r => r.json())

  return (
    <main>
      {services.map((s: Service) => <ServiceCard key={s.id} {...s} />)}
    </main>
  )
}

───────────────────────────────────────────────

// Client Component — interactions, état, événements
'use client'

import { useState } from 'react'

export function AddToCartButton({ articleId }: { articleId: string }) {
  const [added, setAdded] = useState(false)
  return (
    <button onClick={() => setAdded(true)}>
      {added ? 'Ajouté ✓' : 'Ajouter au panier'}
    </button>
  )
}
La règle d'or

Commence toujours en Server Component. Ajoute 'use client' uniquement quand tu as besoin de useState, useEffect, d'event handlers (onClick, onChange) ou d'APIs browser (localStorage, window).

01

Server Actions — mutations sans API

Les Server Actions permettent d'exécuter du code serveur directement depuis un formulaire ou un bouton, sans écrire une API Route séparée. C'est le moyen standard d'envoyer des données en Next.js 15.

Cas concret Formulaire de commande MA-ERI 
// actions/commande.ts
'use server'

interface CommandeData {
  client: string
  email: string
  articles: { id: string; quantite: number }[]
}

export async function envoyerCommande(data: CommandeData) {
  // Validation
  if (!data.email.includes('@')) {
    return { success: false, error: 'Email invalide' }
  }

  // Envoi email via Resend (ou sauvegarde BDD)
  await envoyerEmailConfirmation(data)

  // Revalider le cache Next.js si nécessaire
  revalidatePath('/admin/commandes')

  return { success: true }
}

───────────────────────────────────────────────

// components/FormulaireCommande.tsx
'use client'

import { envoyerCommande } from '@/actions/commande'

export function FormulaireCommande() {
  const [status, setStatus] = useState<'idle' | 'loading' | 'success'>('idle')

  async function handleSubmit(e: React.FormEvent) {
    e.preventDefault()
    setStatus('loading')
    const result = await envoyerCommande({ /* data */ })
    if (result.success) setStatus('success')
  }

  return <form onSubmit={handleSubmit}>...</form>
}
02

API Routes — ton backend dans Next.js

Les API Routes créent des endpoints HTTP dans ton projet Next.js. Elles tournent côté serveur, jamais dans le navigateur — parfait pour les données sensibles.

Exemple Route de contact MA-ERI 
// app/api/contact/route.ts
import { NextRequest, NextResponse } from 'next/server'

// GET /api/contact
export async function GET() {
  return NextResponse.json({ message: 'API MA-ERI opérationnelle' })
}

// POST /api/contact
export async function POST(req: NextRequest) {
  const body = await req.json()

  // Validation du body
  const { nom, email, message } = body
  if (!nom || !email) {
    return NextResponse.json(
      { error: 'Champs requis manquants' },
      { status: 400 }
    )
  }

  // Traitement — envoi email, sauvegarde...
  await sauvegarderContact({ nom, email, message })

  return NextResponse.json(
    { success: true },
    { status: 201 }
  )
}

// Accès : fetch('/api/contact', { method: 'POST', body: ... })
Server Action vs API Route

Server Action : mutations depuis l'UI React (formulaires, boutons). Typé de bout en bout, pas de fetch manuel. API Route : quand tu as besoin d'un endpoint HTTP public — pour un CMS externe, une app mobile, ou un webhook. Pour les features admin MA-ERI, les Server Actions suffisent dans 90% des cas.

Middleware — intercepter les requêtes

Le middleware s'exécute avant chaque requête. C'est là où tu protèges les routes admin, rediriges selon la langue, ou vérifies une session.

Protection Protéger /admin sans backend 
// middleware.ts (à la racine du projet)
import { NextRequest, NextResponse } from 'next/server'

export function middleware(req: NextRequest) {
  const isAdminRoute = req.nextUrl.pathname.startsWith('/admin')

  if (isAdminRoute) {
    const token = req.cookies.get('session')

    if (!token) {
      // Redirection vers login si pas de session
      return NextResponse.redirect(
        new URL('/login', req.url)
      )
    }
  }

  return NextResponse.next()
}

// Appliquer seulement sur certaines routes
export const config = {
  matcher: ['/admin/:path*', '/api/:path*']
}
