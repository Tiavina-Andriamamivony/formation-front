⏱ 2 heures 📍 Débutant React 🛠 React 19 + TypeScript
Chapitre 1

Le problème que React résout

Imagine que tu dois afficher une liste de produits sur un site. En HTML pur, tu écris chaque <div> à la main dans ton fichier. Si tu as 50 produits, tu copies la même structure 50 fois. Et si le design change, tu modifies 50 endroits.

C'est là où React intervient. L'idée centrale est simple : tu décris une fois comment un produit doit s'afficher, et React s'occupe de le répéter autant de fois que nécessaire.

Analogie

Pense à un moule à gâteau. En HTML, tu sculptes chaque gâteau à la main. Avec React, tu crées le moule une fois — et tu peux faire autant de gâteaux que tu veux, tous identiques, juste avec des ingrédients différents.

Ce "moule", dans React, s'appelle un composant. C'est simplement une fonction JavaScript qui retourne du HTML (enfin, quelque chose qui y ressemble très fort — on verra ça juste après).

Chapitre 2

Ton premier composant

Un composant React ressemble à ça :

ServiceCard.tsx tsx 
function ServiceCard() {
  return (
    <div>
      <h3>Approvisionnement industriel</h3>
      <p>Roulements, courroies, joints...</p>
    </div>
  )
}

Tu remarques quelque chose d'étrange ? Il y a du HTML directement dans le JavaScript. C'est ce qu'on appelle le JSX. Ce n'est pas du vrai HTML — c'est une syntaxe spéciale que React transforme en JavaScript avant de l'exécuter.

À retenir

Un composant React = une fonction JavaScript qui retourne du JSX. Le nom commence toujours par une majuscule. C'est tout.

Pour l'utiliser, tu l'écris comme une balise HTML :

App.tsx tsx 
function App() {
  return (
    <div>
      <ServiceCard />
      <ServiceCard />
      <ServiceCard />
    </div>
  )
}

Ça affiche trois fois la même carte. C'est déjà mieux qu'en HTML — mais le problème c'est qu'elles sont toutes identiques. On va régler ça.

Chapitre 3

Les props — personnaliser un composant

Les props (abréviation de "properties") sont les paramètres qu'on passe à un composant. Comme les arguments d'une fonction — parce que c'est exactement ce que c'est.

1
On définit ce que le composant accepte
Avec TypeScript, on écrit une interface — c'est la liste des infos que le composant a besoin pour s'afficher.
2
On utilise ces infos dans le JSX
Entre { }, on peut mettre n'importe quelle expression JavaScript — une variable, un calcul, une condition.
3
On passe les valeurs à l'utilisation
Comme des attributs HTML, mais avec des vraies valeurs JavaScript.
ServiceCard.tsx tsx 
// Étape 1 — l'interface : ce que le composant attend
interface ServiceCardProps {
  titre: string
  description: string
}

// Étape 2 — on utilise les props dans le JSX
function ServiceCard({ titre, description }: ServiceCardProps) {
  return (
    <div>
      <h3>{titre}</h3>
      <p>{description}</p>
    </div>
  )
}

// Étape 3 — on passe les valeurs
<ServiceCard
  titre="Approvisionnement industriel"
  description="Roulements, courroies, joints..."
/>
Pourquoi TypeScript dès maintenant ?

Si tu oublies de passer titre, TypeScript te le dit avant même de lancer le projet. C'est lui qui souligne en rouge dans ton éditeur. C'est ton filet de sécurité — apprends à l'aimer.

Chapitre 4

JSX — les différences avec HTML

JSX ressemble à HTML, mais il y a quelques règles à connaître. Tu vas les rencontrer tout de suite, autant les voir maintenant.

HTML
class="ma-classe"

for="mon-input"

<img src="...">

style="color: red"

<!-- commentaire -->
JSX
className="ma-classe"

htmlFor="mon-input"

<img src="..." />

style={{ color: 'red' }}

{/* commentaire */}

Et une règle importante : un composant ne peut retourner qu'un seul élément racine. Si tu veux retourner plusieurs éléments côte à côte, entoure-les avec un Fragment — ça n'ajoute pas de div inutile dans le HTML final.

Fragment.tsx
// ❌ Erreur — deux éléments au même niveau
return (
  <h1>Titre</h1>
  <p>Texte</p>
)

// ✅ Correct — enveloppé dans un Fragment <>
return (
  <>
    <h1>Titre</h1>
    <p>Texte</p>
  </>
)

Formation 2 sur 4

React fonctionnel —
l'état et les hooks

Tu sais créer des composants. Maintenant on va leur donner de la mémoire — la capacité de se souvenir d'une information et de mettre à jour l'affichage quand elle change.

⏱ 2 heures 📍 Intermédiaire 🛠 React 19 + TS
Chapitre 1

Pourquoi une simple variable ne suffit pas

Avant de voir useState, il faut comprendre pourquoi il existe. Essayons avec une variable normale :

Compteur.tsx
// ❌ Ça ne marchera PAS
function Compteur() {
  let compte = 0

  function incrementer() {
    compte = compte + 1
    // La variable change... mais l'affichage ne bouge pas.
  }

  return (
    <div>
      <p>{compte}</p>
      <button onClick={incrementer}>+1</button>
    </div>
  )
}

Le problème : React ne sait pas que compte a changé. Il n'a aucune raison de ré-afficher le composant. Pour qu'il le sache, il faut lui signaler le changement via un mécanisme prévu pour ça.

Analogie

Imagine que tu modifies un Google Doc hors connexion. Le document change sur ton écran, mais personne d'autre ne le voit. useState, c'est la synchronisation — ça dit à React "hé, quelque chose a changé, tu peux rafraîchir".

Chapitre 2

useState — donner de la mémoire à un composant

useState est un hook — une fonction spéciale fournie par React. Elle retourne deux choses : la valeur actuelle, et une fonction pour la modifier.

Compteur.tsx
import { useState } from 'react'

function Compteur() {
  // [valeur actuelle, fonction pour la changer]
  const [compte, setCompte] = useState(0)
  //                                    ↑ valeur de départ

  return (
    <div>
      <p>{compte}</p>
      <button onClick={() => setCompte(compte + 1)}>+1</button>
    </div>
  )
}

Chaque fois que tu appelles setCompte, React ré-exécute la fonction Compteur avec la nouvelle valeur, et met à jour uniquement ce qui a changé dans le DOM. C'est rapide et automatique.

Nomenclature

Par convention, si la variable s'appelle compte, la fonction s'appelle setCompte. Si c'est isOpen, c'est setIsOpen. Tu verras ce pattern partout.

TypeScript infère le type automatiquement depuis la valeur initiale. Mais parfois tu dois être explicite :

exemples.ts
// TypeScript comprend tout seul que c'est un number
const [compte, setCompte] = useState(0)

// TypeScript comprend tout seul que c'est un string
const [nom, setNom] = useState('')

// Ici il faut être explicite — ça commence à null
// mais ça deviendra un objet Service plus tard
interface Service {
  id: string
  titre: string
}
const [service, setService] = useState<Service | null>(null)
Erreur classique à éviter

Ne modifie jamais un tableau ou un objet directement. liste.push(item) ne déclenchera pas de re-render. Il faut toujours créer une nouvelle valeur : setListe([...liste, item]).

Chapitre 3

useEffect — réagir à des changements

useEffect sert à exécuter du code en réponse à quelque chose — le chargement du composant, ou le changement d'une valeur. C'est là où tu feras tes appels API, par exemple.

Il s'utilise avec un tableau de dépendances qui lui dit quand s'exécuter :

useEffect.tsx
import { useEffect } from 'react'

// Se lance une seule fois, au chargement du composant
useEffect(() => {
  console.log('Le composant vient de s'afficher')
}, []) // ← tableau vide = "une seule fois"


// Se relance à chaque fois que "recherche" change
useEffect(() => {
  fetchResultats(recherche)
}, [recherche]) // ← relancé quand cette valeur change
Pour la suite

En Next.js (formation 3 et 4), tu feras très peu de useEffect pour charger des données. Next.js a une meilleure façon de faire ça côté serveur. Mais useEffect reste utile pour les animations, les timers, et les interactions avec le DOM.

Chapitre 4

Les deux règles des hooks

Les hooks ont deux règles absolues. Les ignorer crée des bugs impossibles à trouver.

1
Toujours en haut du composant
Jamais dans un if, un for, ou une fonction imbriquée. React doit voir les hooks dans le même ordre à chaque render — c'est comme ça qu'il sait lequel est lequel.
2
Seulement dans un composant React
Pas dans une fonction utilitaire normale. Si tu veux partager de la logique avec des hooks, tu crées un custom hook — une fonction dont le nom commence par use.
Les custom hooks

Un custom hook c'est juste une fonction useQuelqueChose() qui utilise d'autres hooks. Ça permet de réutiliser de la logique entre plusieurs composants. On en écrira un dans l'exercice.

Formation 3 sur 4

React vs Next.js —
pourquoi le saut ?

React est une librairie pour construire des interfaces. Next.js est un framework complet bâti sur React. On va comprendre ce que Next.js apporte, et surtout le concept qui change tout : les Server Components.

⏱ 2 heures 📍 Intermédiaire 🛠 Next.js 15
Chapitre 1

Ce que React seul ne fait pas

Quand tu utilises React avec Vite (l'outil de base), tout ton code s'exécute dans le navigateur. Le serveur envoie juste une page HTML presque vide, puis JavaScript charge tout. C'est ce qu'on appelle une SPA — Single Page Application.

Ça pose deux problèmes concrets :

!
Le SEO est cassé
Google reçoit une page vide, puis attend que JavaScript charge. Le contenu n'est pas indexé correctement. Pour un site comme MA-ERI qui doit être trouvé sur Google, c'est un vrai problème.
!
Tout le JS est envoyé au client
Même le code qui lit une base de données, même les clés API secrètes. Il faut jongler avec des serveurs séparés pour tout ça.
Analogie

React seul c'est comme une cuisine qui livre ses repas en ingrédients bruts — le client assemble lui-même. Next.js c'est la cuisine qui envoie le plat prêt à manger. C'est plus rapide à consommer, et Google peut "goûter" directement.

Chapitre 2

Le routing par fichiers

En React classique, tu configures les routes manuellement avec une librairie tierce. En Next.js, la structure de tes fichiers définit tes routes. Pas de configuration.

Structure du projet
app/
├── page.tsx          → la page d'accueil  /
├── layout.tsx        → navbar + footer (commun à toutes les pages)
│
├── services/
│   └── page.tsx      → /services
│
├── produits/
│   ├── page.tsx      → /produits
│   └── [id]/
│       └── page.tsx  → /produits/roulement-6205
│                        /produits/tuyau-acier-50mm
│                        (le [id] est dynamique)
│
└── admin/
    └── page.tsx      → /admin

Tu crées un dossier, tu mets un fichier page.tsx dedans — la route existe. C'est tout.

layout.tsx

Le fichier layout.tsx est un gabarit. Tout ce que tu y mets s'affiche sur toutes les pages en dessous. C'est là que tu mets ta navbar et ton footer — une seule fois pour tout le site.

Chapitre 3

Server Components — le concept central

C'est le concept le plus important de Next.js. Il faut prendre le temps de le comprendre vraiment.

En Next.js, par défaut, tous tes composants s'exécutent sur le serveur. Pas dans le navigateur — sur le serveur. C'est ça un Server Component.

Ça veut dire que tu peux faire des choses impossibles dans un composant React classique :

app/produits/page.tsx
// Ce composant s'exécute sur le serveur.
// Le navigateur reçoit du HTML déjà prêt.

async function ProduitsPage() {

  // On peut faire un fetch directement ici —
  // pas besoin de useEffect, pas besoin d'état "loading"
  const produits = await fetch('https://api.ma-eri.com/produits')
    .then(r => r.json())

  return (
    <main>
      {produits.map((p) => (
        <ProduitCard key={p.id} {...p} />
      ))}
    </main>
  )
}

export default ProduitsPage

Remarque : la fonction est async. Dans un composant React classique, c'est impossible. Ici, c'est naturel — parce que tout se passe sur le serveur, avant que le navigateur reçoive quoi que ce soit.

Chapitre 4

Quand utiliser "use client" ?

Certaines choses ne peuvent pas se faire côté serveur — les interactions, l'état, les événements. Pour ça, tu ajoutes 'use client' en haut du fichier, et le composant redevient un composant React classique côté navigateur.

components/BoutonPanier.tsx
'use client' // ← cette ligne suffit

import { useState } from 'react'

function BoutonPanier({ produitId }: { produitId: string }) {
  const [ajoute, setAjoute] = useState(false)

  return (
    <button onClick={() => setAjoute(true)}>
      {ajoute ? 'Ajouté ✓' : 'Ajouter au panier'}
    </button>
  )
}
La règle simple

Commence toujours sans 'use client'. Ajoute-le uniquement quand TypeScript ou React te dit que tu ne peux pas utiliser useState, useEffect, ou un gestionnaire d'événement (onClick, onChange...) sans ça.

Server Component
✓ async/await
✓ fetch de données
✓ accès fichiers
✓ SEO parfait
✗ pas de useState
✗ pas de onClick
Client Component
✓ useState
✓ useEffect
✓ onClick, onChange
✓ animations
✗ pas de async direct
✗ JS envoyé au client

Chapitre 1

Server Actions — envoyer des données sans API

Jusqu'ici on a vu comment afficher des données. Mais comment on envoie des données ? En React classique, tu créerais une API, ferais un fetch POST, gèrerais les erreurs... Beaucoup de code pour quelque chose de simple.

Next.js 15 a une solution : les Server Actions. C'est une fonction qui s'exécute côté serveur, mais que tu appelles directement depuis ton composant — comme si c'était une fonction locale.

1
Tu crées la fonction serveur
Elle commence par 'use server' — c'est la directive qui dit à Next.js "ce code ne va jamais dans le navigateur".
2
Tu l'importes dans ton composant
Comme n'importe quelle fonction. Next.js gère tout le transport automatiquement.
3
Tu l'appelles comme une fonction normale
Mais en coulisses, Next.js envoie une requête au serveur, exécute le code, et retourne le résultat.
actions/commande.ts
'use server'

export async function envoyerCommande(data: {
  client: string
  email: string
}) {
  // Ce code s'exécute sur le serveur.
  // On pourrait lire une base de données, envoyer un email...
  // Le navigateur ne verra jamais ce code.

  if (!data.email.includes('@')) {
    return { ok: false, erreur: 'Email invalide' }
  }

  // ... traitement ...

  return { ok: true }
}
components/FormulaireCommande.tsx
'use client'

import { useState } from 'react'
import { envoyerCommande } from '@/actions/commande'

function FormulaireCommande() {
  const [statut, setStatut] = useState<'idle' | 'loading' | 'ok'>('idle')

  async function handleSubmit() {
    setStatut('loading')
    const resultat = await envoyerCommande({
      client: 'Jean Rakoto',
      email: 'jean@example.mg'
    })
    if (resultat.ok) setStatut('ok')
  }

  return (
    <button onClick={handleSubmit} disabled={statut === 'loading'}>
      {statut === 'loading' ? 'Envoi...' : 'Commander'}
    </button>
  )
}
Pas de backend séparé

Pour les features MA-ERI sans backend, les Server Actions sont parfaites. Tu peux envoyer un email de confirmation, sauvegarder dans un fichier, ou appeler une API externe — tout ça sans jamais créer de serveur séparé.

Chapitre 2

API Routes — créer un endpoint HTTP

Parfois tu as besoin d'un vrai endpoint HTTP — pour un webhook, pour une app mobile, ou pour un service externe qui doit appeler ton site. C'est là qu'on utilise les API Routes.

Un fichier route.ts dans le dossier app/api/ crée automatiquement un endpoint.

app/api/contact/route.ts
import { NextRequest, NextResponse } from 'next/server'

// Répond aux requêtes POST sur /api/contact
export async function POST(req: NextRequest) {
  const { nom, email, message } = await req.json()

  if (!nom || !email) {
    return NextResponse.json(
      { erreur: 'Champs manquants' },
      { status: 400 }
    )
  }

  // Traitement...

  return NextResponse.json({ ok: true })
}
Server Action ou API Route ?

Tu appelles ça depuis ton UI React ? → Server Action. Un service externe doit t'appeler par HTTP ? → API Route. Dans le doute, Server Action — c'est plus simple.

Chapitre 4

Middleware — protéger des pages

Le middleware est un fichier spécial qui s'exécute avant chaque requête. C'est là où tu protèges les pages qui ne doivent pas être accessibles à tout le monde — comme ton interface admin.

middleware.ts (à la racine)
import { NextRequest, NextResponse } from 'next/server'

export function middleware(req: NextRequest) {
  const estAdmin = req.nextUrl.pathname.startsWith('/admin')

  if (estAdmin) {
    const session = req.cookies.get('session')

    // Pas de session → on redirige vers /login
    if (!session) {
      return NextResponse.redirect(
        new URL('/login', req.url)
      )
    }
  }

  // Tout va bien → on laisse passer
  return NextResponse.next()
}

// Sur quelles routes ce middleware s'applique
export const config = {
  matcher: ['/admin/:path*']
}
Pour l'interface admin MA-ERI

La protection par middleware sera la première chose à mettre en place quand tu commenceras la feature admin. Personne ne doit pouvoir accéder à /admin sans s'identifier.
